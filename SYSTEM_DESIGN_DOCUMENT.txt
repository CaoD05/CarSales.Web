1.3 TECHNOLOGY STACK DETAILS

Framework:          .NET 10
Web Framework:      ASP.NET Core (Razor Pages + MVC Controllers)
Database ORM:       Entity Framework Core 10.0.5
Database:           SQL Server
Session Management: Distributed Memory Cache with Session Middleware
API Documentation: Swagger (Swashbuckle.AspNetCore 10.1.5)
Authentication:    Session-based (Custom attribute-based authorization)


1.4 KEY ARCHITECTURAL PATTERNS

1. Layered Architecture
   - Presentation Layer: Razor Pages Views and API Controllers
   - Business Logic Layer: Controllers and API Endpoints
   - Data Access Layer: Entity Framework Core with DbContext
   - Database Layer: SQL Server

2. MVC Pattern
   - Models: Entity classes representing domain objects
   - Views: Razor templates for rendering HTML
   - Controllers: Business logic and request handling

3. Repository-like Pattern
   - ApplicationDbContext serves as the data access abstraction layer
   - Direct DbSet access through DbContext

4. Attribute-Based Authorization
   - Custom [RoleAuthorize] attribute for role-based access control
   - Session-based user context storage

5. API-Driven Frontend
   - Separate API controllers for data operations
   - Frontend can consume REST endpoints
   - Swagger UI for API documentation

2.1 AUTHENTICATION & USER MANAGEMENT

Components:
  - AccountController.cs
  - AuthApiController.cs
  - User.cs (Model)
  - LoginViewModel.cs / RegisterViewModel.cs
  - RoleAuthorizeAttribute.cs

Features:
  • User Registration
    - Email validation
    - Password hashing using PasswordHelper
    - Role assignment (Admin, Staff, Customer)
    - Profile creation (Full Name, Phone, Address, Avatar)

  • User Login
    - Email-based authentication
    - Session management (30-minute idle timeout)
    - Role-based redirect to appropriate dashboard

  • User Profile Management
    - Update profile information
    - Avatar upload support
    - Contact details management

  • Role-Based Access Control (RBAC)
    - Three role types: Admin, Staff, Customer
    - Custom RoleAuthorizeAttribute enforces role checks
    - Session-based role validation
    - Role hierarchy: Admin > Staff > Customer

Workflow:
  1. User registers with email and password
  2. Password is hashed before storage
  3. User logs in with email credentials
  4. Session stores UserId and Role
  5. RoleAuthorizeAttribute validates access based on stored role
  6. Session expires after 30 minutes of inactivity

5. SECURITY FEATURES
================================================================================

5.1 AUTHENTICATION & AUTHORIZATION

  • Session-Based Authentication
    - 30-minute idle timeout
    - HttpOnly cookies (prevents JavaScript access)
    - Secure flag for HTTPS transmission

  • Password Security
    - Hashed passwords using PasswordHelper
    - Never stored in plain text
    - Salt applied during hashing

  • Role-Based Authorization
    - Custom RoleAuthorizeAttribute
    - Role validation on every protected request
    - Three role levels: Admin, Staff, Customer

  • Session Validation
    - UserId and Role stored in session
    - Checked before granting access
    - Invalid sessions redirect to login


5.2 DATA PROTECTION

  • Foreign Key Constraints
    - Prevent orphaned data
    - OnDelete behavior: Restrict (prevents accidental data loss)

  • Input Validation
    - Required field validation
    - Email format validation
    - Phone number validation
    - String length constraints

  • SQL Injection Prevention
    - Entity Framework Core parameterized queries
    - No dynamic SQL construction


5.3 API SECURITY

  • CORS Headers
    - Controlled cross-origin requests

  • Swagger Documentation
    - Available in development mode only

  • Session Token Management
    - Session ID validation
    - CSRF protection via session state

7. PERFORMANCE CONSIDERATIONS
================================================================================

7.1 OPTIMIZATION TECHNIQUES

  • Entity Framework Include Statements
    - Eager loading to prevent N+1 queries
    - Example: Include(x => x.CarType) for car listings

  • Pagination
    - Large result sets handled with pagination
    - Limits database query results

  • Caching
    - Distributed Memory Cache available
    - Can cache frequently accessed data (brands, types)

  • Session Caching
    - User session data stored in memory
    - Reduces authentication overhead


7.2 SCALABILITY CONSIDERATIONS

  • Stateless API Design
    - API endpoints can be load-balanced
    - Session state in memory (can be moved to Redis for distributed)

  • Database Connection Pooling
    - SQL Server connection pooling
    - Entity Framework handles connection management

  • Static File Serving
    - wwwroot files can be served by CDN
    - Reduces application server load
